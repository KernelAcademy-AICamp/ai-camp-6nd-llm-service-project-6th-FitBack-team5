# FitBack 컨셉 정의 — AI 페이스메이커 "선생님" (2026-06-15)

> 핵심 컨셉: **AI 페이스메이커 = 끝까지 데려가는 선생님 / 회원권 본전 / 직장인 친근(비위협·동행)**
> 본 문서는 방향 정리용. 코드 인벤토리(사실)와 코치 피드백을 종합했고, 미확정은 "제안/검토"로 표시. 네이밍은 후속(보류).

---

## 배경 — 왜 이 컨셉이 나왔나 (도출 과정)

> 이 기능은 새 아이디어가 아니라 **MVP 점검 → 지표 분석 → 코치 피드백 → 갭 진단**을 거쳐 *수렴*된 결과다.

1. **출발점(원래 페인포인트)** — "결제한 회원권을 끝까지 못 쓴다"는 문제. 이를 풀려고 MVP로 **활용도 가시화·체크인·기록·식단·운동·코치 카드**를 구축했다.
2. **MVP 점검에서 드러난 한계** (`mvp-review-0615.md`)
   - 핵심 길은 동작하나, **무료권 활용도 미작동·운동 AI 키 노출** 등 결함.
   - **AI가 비핵심(식단 4·운동 2)에 쏠리고**, 정작 핵심인 *행동 유도 AI는 코치 카드 1종*뿐 → ROI 정렬 역전.
3. **북극성 지표(NSM) 분석** — 진짜 가치 순간 = **검증된 체크인**. 그런데 ① 베타에서 측정이 어렵고 ② 더 근본적으로 **"체크인을 일으키는" 장치가 약하다**는 걸 확인.
4. **팀·코치 논의**
   - 팀: post-MVP 방향(웰니스 확장? AI 코치 강화? 타겟 좁히기?).
   - **마케팅 코치**: "고객 니즈 = 프로덕트", *기능 발산("서비스에 치중") 경고*, 경험 중심.
   - **실습 코치**: 선생님 기능 확장은 문서로 정리(사용자 테스트 제외) + **AI 봇 테스트**.
5. **핵심 진단(전환점)** — 현재 기능은 *"이미 가기로 한 사람"*의 마찰만 줄인다. **"안 가던 직장인을 그 순간 가게"** 만드는 **트리거·적응형 코칭·재활성 루프가 공백** = NSM(체크인)의 진짜 병목.
6. **그래서 도출** — 단발 코치 카드를 넘어 **끝까지 데려가는 AI 페이스메이커(선생님)**로 진화.
   - 핵심 가치(**회원권 본전**)에 *수렴* → 마케팅 코치의 "발산 금지"와 정합.
   - **NSM(체크인)을 직접 견인** → 성장 레버.
   - "고객이 프로덕트 안에서 즐겁게"(관계·동행) → 마케팅 코치의 경험 요구 충족.
7. **타겟을 좁힌 이유** — 직장인 20후~30중은 출석 장벽(시간·피로·미룸)이 가장 선명해, 트리거·적응형 코칭의 효과와 메시지가 또렷해진다.

**요약**: *"활용도를 보여주는" MVP만으로는 안 가던 사람이 가지 않는다 → "끝까지 데려가는 선생님"이 빠진 조각이라는 결론.*

## 0. 한 줄 컨셉 & 피치
- **컨셉**: 끊어두고 안 가던 직장인을, AI가 **페이스메이커(선생님)**처럼 **끝까지 데려가** 회원권 본전을 뽑게 한다.
- **피치 한 줄**: *"끊고 안 가던 직장인을, AI 선생님이 끝까지 데려가 회원권 본전을 뽑게 한다."*

## 1. 타겟
- **주 타겟**: 직장인 **20대 후반 ~ 30대 중반**. (기존 "여성 20–35" → 직장인으로 더 좁힘)
- 특징: 야근·불규칙 퇴근, 퇴근 후 방전, "끊어놓고 못 가는" 죄책감. '몸짱'보다 **컨디션·번아웃 관리** 동기가 큼.

## 2. 포지셔닝 — 3층 분리 (헬스 ↔ 웰니스 충돌 해소)
| 층 | 내용 | 헬스/웰니스 | 불변 |
|---|---|---|---|
| **WHY (감정·간판)** | 지치지 않고 꾸준한 직장인의 컨디션 | **웰니스 톤** | 넓혀도 됨 |
| **WHAT (JOB·약속)** | 끊은 회원권을 끝까지 쓰게 | **헬스(회원권)** | **고정** |
| **NSM (측정)** | 검증된 주간 체크인 | **헬스** | **고정** |
| **HOW (선생님)** | 컨디션 읽고 → 행동(체크인)으로 연결 | 웰니스→헬스 | — |

- **결론**: "웰니스 선생님"으로 **부르되**, 임무·측정은 **회원권 본전·체크인에 못 박는다**. 전면 웰니스 전환 ✕(앵커·NSM 상실, 마케팅 코치의 "발산" 경고).
- **웰니스 범위 경계(중요)**: *출석을 돕는 컨디션*만 — 운동 후 회복, 식단=연료/회복, 피로·일정 고려 페이스 조정. **제외(MVP)**: 수면 트래킹·명상·스트레스 측정·범용 건강관리. 판단 기준 = *"이 기능이 체크인(NSM)을 올리나?"*

## 3. "선생님" 개념 (IS / IS NOT)
**선생님 = 컨디션을 읽고, 끊은 회원권을 끝까지 쓰도록 옆에서 끌고 가는 책임 코치(동행자).**

| 선생님이다 ⭕ | 선생님이 아니다 ✕ |
|---|---|
| 컨디션·일정 고려해 *이번 주 어떻게 갈지* 같이 짜줌 | 운동 시키고 평가하는 트레이너/감독 |
| 못 간 날도 포기 않고 보충안 제시(동행) | 한 번 조언하고 마는 단발 정보봇 |
| 회원권 본전이라는 구체 목표로 데려감 | 수면·명상까지 다 다루는 범용 웰니스 챗봇 |
| 성취를 같이 기뻐함(관계·재미) | 죄책감·압박을 주는 존재 |

**선생님 행동 4단계 (닫힌 루프)**: ① 관찰(페이스·방문·컨디션) → ② 처방(이번 주 플랜) → ③ 동행(못 간 날 보충 nudge → 체크인 딥링크) → ④ 조정(지난주 보고 다음 주 난이도·요일 조정).

## 5. 현재 기능으로 풀리는가 — 갭 분석 (코드 인벤토리 기준)
| 장벽 | 현재 기능 | 해결도 |
|---|---|---|
| B5 성취 | 활용도·스트릭·주간 기록·체크인 즉시 본전↓ | 🟢 양호 |
| B6 이동 | 체크인 길안내(지도·대중교통·날씨) | 🟢 양호 |
| B1 결정 | 권장 페이스(**고정 주2회**)·코치 단발 추천 | 🟡 부분(내 일정 반영 ✕) |
| B4 행동 | 코치 추천 + 홈트 대안 | 🟡 부분(앱 열었을 때만) |
| B2 컨디션 | 없음 | 🔴 공백 |
| B3 트리거 | **알림 미포함**(홈 배너 대체) | 🔴 공백 |

**결론**: 현재 기능은 *"이미 가기로 한 사람"의 마찰·성취*는 잘 푸나(B5·B6), *"안 가던 사람을 그 순간 가게"* 만드는 **트리거(B3)·적응형 코칭(B1/B2)·재활성 루프**가 비어 있다. = 피치의 "끝까지 데려가" 핵심이 미구현.

### 현재 비어 있는 것 (선생님 v1 과제)
1. **트리거(개입 타이밍)** — 알림/약속/캘린더형 장치 부재 (가장 큰 공백)
2. **적응형·일정/컨디션 인지** — 페이스 상수, 코치 6h 단발 → 야근·피로 모름
3. **회복·식단의 출석 결속** — 식단 AI 독립, 출석 루프와 미연결
4. **재활성(리인게이지) 루프** — "안 가던 사람" 다시 끌어오는 흐름 없음

## 6. 지표 연결 (NSM 고정)
- **북극성(L0)**: 주간 **검증 체크인 수**. 포지셔닝을 웰니스로 넓혀도 **NSM은 체크인 유지**(웰니스 점수로 바꾸면 허영 지표).
- **선생님이 직접 키우는 L1**: 코치 추천 → 체크인 전환율 / 첫 체크인 전환율 / 주간 재방문율.
- 즉 **선생님 강화 = NSM 성장 레버**. 웰니스 확장은 NSM과 약결합이라 단독 추진 ✕.

## 7. 마케팅 코치 렌즈 (피칭 정합)
- **고객 혜택**: 돈(회원권 손실 방지) + 건강(꾸준함) — 이중 가치, 측정 가능.
- **수익(SaaS)**: B2C 구독(선생님 프리미엄) **또는** B2B(헬스장 제휴: 재등록률↑=헬스장 이익) → **검토 필요**.
- **팀 자산**: 방문·행동·코칭 반응 데이터 = 적응형 코칭의 해자.
- **경험**: "고객이 프로덕트 안에서 즐겁게" = 선생님과의 관계·성취 경험(쿠팡/마컬식 락인).

## 8. 실습 코치 가이드 반영
- **문서화**: 현재까지 피봇 + 선생님 확장 로드맵 정리(본 문서). **선생님 기능은 사용자 테스트 범위 제외**.
- **테스트 분리**: 현재 MVP(핵심 길·체크인) = 사용자 테스트 / **선생님 = AI 봇으로 별도 테스트**.

## 9. 결정 필요 (검토)
1. **타겟 공식 확정**: 직장인 20후~30중.
2. **웰니스 경계**: §2 "출석을 돕는 컨디션만" 동의 여부.
3. **트리거 방식(최우선)**: 푸시 알림 재검토 vs 약속/예약형(앱 내) vs 캘린더 연동. ※ 팀 결정 4(알림 MVP 미포함)와 충돌 → 출석을 만들려면 재검토 필요.
4. **수익 모델 1순위**: B2C 구독 vs B2B 헬스장 제휴.
5. **선생님 v1 범위**: (a)트리거 / (b)적응형 플랜 / (c)보충 루프 중 우선.

- **준영(뭐 할지+진전)**: "2주간 가슴·등만, 하체 0"(관찰) → "오늘 하체? 부담되면 다른 부위"(처방, 단일 부위+거절) → 기록 후 "상체4·하체1, 회당 ₩40k→₩37k, ₩3k 회수"(동행) → "하체 한 번 더 끼울까요"(조정)
- **하늘(재활성)**: 3주 미방문 "바쁘셨죠, 41일 남았어요"(관찰) → "이번 주 딱 1번"(처방, 장벽↓) → "주말 컨디션 괜찮은 날 하루만"→1회 복귀(동행) → "다음 주도 1회부터"(조정)
- **서연(마찰 낮추기)**: "짐부터 같이 — 타올·운동복만"(처방, 행동 쪼갬) → "지금 출발 도보 12분, 비 안 와요 [출발하기]"→1km 자동 체크인(동행) → "준비물 미리, 아침에 알려줄까요"(조정, 마찰 사전이동)
- **도윤(예방·습관)**: 온보딩 직후 시드로 홈 안 빔 "첫 주는 2번만"(관찰) → "가능 수업 화19·금19, 둘 다?"(처방) → Day1~5 고정 "첫 방문 완료! 회당 첫 하락"(동행, Aha) → 6일째 "화·토로 굳힐까요"(개인화 전환)

## 12. 경쟁사 검토 & 포지셔닝

- **전제**: 국내 AI 챗봇 경쟁사는 대부분 식단·칼로리·혈당에 몰려 있고 동기 훅이 "감량/건강". **"회원권 본전(활용도)" 훅 + 출석 행동 유도에 챗봇을 붙인 사례는 사실상 없음**(구매 후 미활용 공백과 일치). 챗봇 3대 기능 자체는 시장 성숙 → 신규성보다 **"본전 맥락에 묶는 방식"**이 차별점.
- **챗봇 3대 기능**: ① 프롬프트→플랜/식단 생성 + 음식 사진 분석 · ② 질문지 가이드 제안 · ③ 기록물 재활용
- **기능별 시사점**
  - ① 사진→칼로리/영양 분석은 표준(필라이즈·파스타 등) → 신규성 약함. **생성물을 본전 계산·체크인과 연결**하는 데 비중.
  - ② 질문지 가이드는 일부만 구현(인아웃 추천 질문, 다톡이 능동 질문). **선생님이 컨디션·일정을 묻고 주간 플랜을 짜는 흐름은 경쟁 강도 낮음** → 여지 있음.
  - ③ 기록물 재활용은 사례 존재(다톡이 정체기 분석, 필라이즈 전날 기반 미션). **FitBack 차별 = 챗봇 생성물(플랜·식단)을 대시보드·체크인·본전 계산에서 구조적으로 재사용.**
- **종합 포지셔닝**: 차별화는 기능이 아니라 (a) 동기 훅을 "감량"이 아닌 **"회원권 본전"**으로, (b) 출석·체크인 행동으로 수렴시키는 **"선생님" 루프**, (c) **챗봇 생성물의 타 기능 재활용 구조**에서 나옴. 식단/혈당 중심 챗봇과 직접 충돌을 피하고, **식단·운동 플랜을 "본전 회수" 목적함수에 종속**시키는 포지셔닝이 안전.

## 13. 대시보드 지표 방향 — "쌓이는 절약액(상승)"

- **현재 문제(회당 비용 하락 표시)**: ① 방향이 직관과 반대(올라야 성취) ② 보상이 갈수록 작아짐(방문 쌓일수록 하락 폭 미미 → 잘할수록 보상 약화) ③ 초반 절대값이 위협적 → "동행·비위협" 원칙과 충돌.
- **개선 방향**
  - **히어로 = 누적 구한 돈(상승)**: "오늘 체크인으로 ₩6,800 구함 · 이번 달 누적 ₩27,000". 방문할수록 커져 보상 강화(음수 미표시 원칙 합치).
  - **정가 앵커 병기**: "1회 정가 ₩25,000 → 지금 ₩18,200에 이용 중"(준거점 효과).
  - **회당 비용은 보조로 강등**: 없애지 말고 상세/툴팁에 "현재 회당 ₩18,200".
- **프레임 분리**: 체크인 **전 = 손실 넛지**("오늘 가면 ₩6,800 살릴 수 있어요") / **후 = 이득 보상**("₩6,800 구했어요"). 계산은 기존 본전/baseline 로직(결제액÷20 등) 재사용, **표시만 상승 누적값**으로.
- ⚠️ **구현 영향**: 현재 [index.tsx](src/app/index.tsx) 히어로는 "회당 비용 + ↓" 프레임 → 이 방향에 따라 **"누적 절약액(상승)"으로 변경 대상**(부록 §B 및 구현 갱신은 후속 작업). NSM(검증 체크인)은 불변.

### A-2. 베타에선 "대리 지표(Process)"로 측정

**왜 베타에서 진짜 NSM을 못 잡는가**
| 이유 | 설명 |
|---|---|
| 저빈도 outcome | 방문 주 1~2회 → 5일·소수 N에선 표본 부족, 통계 의미 상실 |

| 기능 사용성과 무관 | "방문 수"는 outcome일 뿐, 체크인 플로우 사용성·코치 클릭은 별개 측정 |

**해결책: 두 단계 측정 분리**
- **베타(5일)** — 목표=기능 사용성 검증 / 측정=process 지표(완주율·클릭율) / 진짜 NSM=directional만
- **Post-Beta(코호트)** — 목표=진짜 NSM 본격 측정 / 측정=outcome NSM(검증 방문)+충분한 N

**베타 측정 설계 (4트랙)**

**① 대리 NSM — 체크인 완주율 (정량)**
| 지표 | 정의 | 측정법 | 베타 목표 |
|---|---|---|---|
| 체크인 완주율 | 완료÷시작 | 이벤트 | 기준선 후 목표 설정 |

| ① 온보딩 완료율 | 단계 축소·이탈 단계 개선 | 완료÷가입 (%) | FE(온보딩) |
| ② 첫 체크인 전환율 | 첫 방문 유도·코치 첫 처방·시드 데이터(빈 홈 방지) | 7일내 첫 체크인÷온보딩완료 (%) | PM+FE(체크인) |
| ③ 코치→체크인 전환율 | 적시 트리거(알림/푸시)·누적 절약액 보상 | 72h내 체크인÷트리거 노출 (%) | FE/BE(코치) |
| ④ 주간 재방문율(W/W) | 개인화 페이스·보충 nudge·재활성(3주 미방문) | 지난주+이번주÷지난주 (%) | PM/Growth |

| 지표 | 정의 | 경고 |
|---|---|---|
| 자기신고 방문 비율 | 자기신고÷전체 방문 | ↑이면 지오펜스 신뢰 저하 → NSM 신뢰도 하락 |

## C. UX 이슈 → 가설 변환 (5가지)

가설 공식: **[개입]을 하면 [타겟]의 [지표]가 [방향]할 것이다 — [이유]. (측정: 지표)**

| # | 이슈 | 가설 | 이유 | 측정 |
|---|---|---|---|---|
| 1 | 회당 비용 "하락"만 표시 → 보상 약함 | "오늘 ₩6,800 절약·이번 달 누적 ₩27,000"(상승)으로 바꾸면 주간 재방문율↑ | 방문할수록 누적액↑, 강화 신호 누적 | 주간 재방문율 |
| 2 | 출석 트리거 부재(앱 안 열면 계기 X) | 적시 트리거(약속/알림)면 코치→체크인 전환율↑ | "그 순간" 미루는 장벽 직접 깸 | ③ 코치→체크인 전환율 |
| 3 | 권장 페이스 고정(주2회), 야근·컨디션 무시 | 일정·컨디션 반영 개인화 페이스면 주간 재방문율↑ | 직장인은 고정 권장 무시, 현실적 제안 필요 | ④ 주간 재방문율 |
| 4 | 재활성 흐름 없음(3주 미방문 방치) | 미방문 감지+"이번 주 1번만" 부담↓ 처방→복귀율↑ | 진입 장벽 낮추면 다시 오기 시작 | ④ 재방문율(복귀자 서브그룹) |
| 5 | 이동·준비 마찰(큰 결심 필요) | 마이크로 스텝("타올·운동복만"+"지금 출발 도보 12분")→첫 체크인 완주율↑ | 큰 행동을 잘게 쪼개면 시작 용이 | 체크인 완주율+단계별 이탈 |

## D. AI 활용 (구체화)

**원칙: 신뢰 > 기술** — 회원권 본전·절약액·NSM 계산은 **비-AI 결정론적 산식**. 사용자가 "왜 이 숫자?"를 신뢰해야 하므로, **AI는 감정·추천만** 담당.

| 기능 | 담당 모델 | 역할 |
|---|---|---|
| 1. 주간 코치 처방 | Claude Opus / Sonnet | 지난주 운동 패턴 분석 → 이번주 추천 운동 + 개인화 격려 |
| 2. 식단 분석 | Claude Vision | 식사 이미지 → 추정 칼로리·영양 분석 |
| 3. 실시간 피드백 | Claude Haiku | 빠른 응답이 필요한 짧은 피드백/격려 |
| 4. 보상·절약액 계산 | Python / SQL | 비-AI 결정론적 산식(신뢰 우선) |

**거버넌스**: 현재=개별 API key(feature branch) → 메인=중앙 key(GitHub Secrets) → Post-MVP=운동 AI 키 서버 이관 후 모든 호출을 Edge Function 프록시로 통일(보안·감시·비용 제어).
